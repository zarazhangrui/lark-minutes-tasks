读取飞书妙记，提取行动项，然后直接帮用户把事情办了（从 to-do 到 done）。

输入：$ARGUMENTS（飞书妙记 URL、minute_token，或会议纪要文档 URL）

## 核心理念

不要只给用户列一堆待办。目标是直接替用户执行，把 to-do 变成 done。比如：

- 会上说"要给谁发一个东西" → 直接替用户发
- 会上说"回头试用一下某产品" → 找到产品链接发给用户提醒
- 会上说"跟 XX 约个 follow up" → 直接查日历空闲时间、帮用户把会约了
- 会上说"把这个反馈整理一下发给某某" → 整理好内容直接发
- 会上说"看一下某个文档" → 把文档找到、读完、摘要发给用户

**所有执行动作之前必须先找用户确认。** 先展示完整的行动计划，用户说"做吧"再动手。

## Wake Word 功能

用户可以在开会时通过说出 wake word 来直接给 Agent 下指令。这不是实时触发的（妙记逐字稿是会后生成的），而是在分析逐字稿时自动扫描识别。

**工作原理：** 在第四步分析逐字稿时，扫描当前用户的所有发言，寻找包含 wake word 的句子。wake word 后面的内容视为用户直接给 Agent 的指令，自动提取为最高优先级行动项，在 HTML 界面中置顶显示并预设状态为"交给 Agent"。

**Onboarding 配置（首次使用时）：** 首次使用 skill 时，需要完成以下配置，记录到 memory 中，后续不再重复询问：

1. **用户身份**：通过 `lark-cli api GET /open-apis/authen/v1/user_info --as user` 自动获取当前用户姓名（用于在逐字稿中匹配发言人），无需手动询问
2. **Wake word**：询问用户想设置什么 wake word（比如"帮我"、"Hey assistant"等，任何词都行），默认为空（不启用）

**识别规则：**

- 只扫描用户本人的发言（不扫描其他参会人的发言）
- wake word 不区分大小写
- wake word 后面到该段发言结束（或到下一个句号/逗号自然断句）的内容作为指令内容
- 如果 wake word 出现在明显不是指令的上下文里（比如在讨论某个产品或技术本身），用常识判断跳过

**示例：**

- 用户说"[wake word]，帮我约一下小王下周聊" → 提取为行动项："约小王下周聊"，类型：约会议
- 用户说"[wake word]，回头把这个方案整理成文档发给老板" → 提取为行动项："整理方案文档发给老板"，类型：写文档+发消息
- 用户说"我觉得 XX 的模型能力还不错"（wake word 出现在产品讨论中）→ 跳过，这不是指令

## 执行步骤

### 第一步：解析输入和 wake word 配置

检查 memory 中是否已有用户配置（姓名 + wake word）。如果没有，进行 onboarding：

1. 自动调用飞书 API 获取用户姓名（`lark-cli api GET /open-apis/authen/v1/user_info --as user`）
2. 询问 wake word 设置（默认为空，不启用）

记录到 memory 后继续。如果已有配置则直接使用。

根据输入 URL 的格式判断类型，走不同的路径：

**路径 A：妙记 URL（`/minutes/` 路径）**

- 格式举例：`https://bytedance.us.larkoffice.com/minutes/obusnm4346glpgcs688ao317`
- 取路径最后一段作为 minute_token，忽略 query 参数
- 直接进入第二步

**路径 B：飞书文档 URL（`/docx/` 或 `/wiki/` 路径）**

- 格式举例：`https://bytedance.us.larkoffice.com/docx/Gihmd4ptqoTL5HxKjaLus6RMsmh`
- 这是会议纪要文档，不是妙记本身。处理方式：
  1. 用 `docs +fetch` 读取文档内容
  2. 在文档末尾寻找逐字稿链接。注意：逐字稿通常也是一篇飞书文档（`/docx/` 链接），不是妙记链接
  3. 如果找到逐字稿文档链接，再用 `docs +fetch` 读取逐字稿文档的内容，然后按正常流程分析逐字稿
  4. 如果找到妙记链接（`/minutes/`），提取 minute_token，走路径 A
  5. 如果都没找到，就直接基于会议纪要文档本身的内容来提取行动项（文档里通常已经有 AI 生成的总结、待办、章节等内容，可以直接分析）

### 第二步：获取妙记元信息

```bash
lark-cli minutes minutes get --params '{"minute_token": "TOKEN"}'
```

记录标题、时长、参会人信息。

### 第三步：获取逐字稿和 AI 产物

```bash
lark-cli vc +notes --minute-tokens TOKEN
```

这会下载逐字稿文件并返回 AI 总结、章节、待办等。读取返回的 transcript.txt 完整内容。

### 第四步：深度分析逐字稿

逐字稿是最核心的输入。仔细阅读全文，提取以下内容：

**0. Wake word 指令扫描**（最先做！）

先扫描用户本人的所有发言，找出包含 wake word 的句子。这些是用户在会上直接给 Agent 的指令，优先级最高。提取出来后单独标记为"用户直接指令"，在行动项列表中置顶显示。

**1. 可执行的行动项**（这是唯一的输出内容，不需要会议摘要、关键决策、未解决问题等，那些属于妙记本身的功能。注意不要和 wake word 指令重复）

对每一条行动项，判断它属于哪种类型，并制定执行方案：

| 行动类型          | 识别信号                               | 执行方案                                                         |
| ----------------- | -------------------------------------- | ---------------------------------------------------------------- |
| 发消息/转发内容   | "把这个发给XX"、"告诉XX"、"跟XX说一下" | 起草消息内容，复制到剪贴板，让用户一键粘贴发送                   |
| 约会议/follow up  | "约个时间"、"下周聊"、"follow up"      | 用日历 +suggestion 查空闲时间，+create 创建日程                  |
| 写文档/整理内容   | "整理一下"、"写个文档"、"总结一下"     | 用 docs +create 创建飞书文档                                     |
| 查信息/调研       | "看一下"、"了解一下"、"调研"           | 用 WebSearch/WebFetch 查找信息，或用飞书 docs +search 找内部文档 |
| 试用产品/工具     | "试一下"、"体验一下"、"装一下"         | 找到产品链接/安装方式，发给用户                                  |
| 创建任务/分配工作 | "安排一下"、"分配给XX"                 | 用飞书任务创建待办                                               |
| 读文档/review     | "看一下这个文档"、"review 一下"        | 找到文档，读完后给用户摘要                                       |
| 其他/需人工处理   | 复杂决策、线下活动等                   | 标记为"需要你亲自处理"，说明原因                                 |

每条行动项标注：

- 负责人（谁说的、谁应该做）
- 具体事项
- 我打算怎么帮你执行
- 截止时间（如果提到的话）

### 第五步：展示编号列表并等待用户指令

把所有行动项以清晰的编号列表展示在对话中。格式：

```
## 行动项（共 N 项）

1. [标题]
   说明：[描述]
   方案：[Agent 打算怎么执行]

2. [标题]
   说明：[描述]
   方案：[Agent 打算怎么执行]
   需要补充：[如果需要用户提供信息]

...
```

然后告诉用户可以用编号快速回复，比如：

- "1 3 5 你来做" 或 "do 1 3 5"
- "2 已经做了"
- "4 改成：把文档发给 A 而不是 B"
- "全部你来做"
- "除了 6 7 其他都你来做"

用户不需要打完整的句子，只要用编号指明哪些项 + 动作就行。

### 第六步：执行

用户确认后，逐条执行。每完成一件，报告结果。可能用到的工具：

- **发消息**：起草消息内容，用 pbcopy（macOS）或类似工具复制到剪贴板，告知用户去哪个聊天窗口粘贴发送。不要代替用户发送。
- **约会议**：lark-cli calendar +suggestion → +create
- **写文档**：lark-cli docs +create
- **查文档**：lark-cli docs +search → docs +fetch
- **搜索信息**：WebSearch / WebFetch
- **创建任务**：lark-cli task 相关命令
- **找人**：lark-cli contact +search-user

如果某个执行步骤因为权限或 API 限制做不了，说明卡点，提供 workaround（比如复制内容到剪贴板）。

## 注意事项

- 逐字稿里的发言人名字可能是拼音或英文名，尽量匹配到真实姓名
- 时间戳格式是"分:秒"，引用时写成"XX分XX秒"方便阅读
- 行动项要穷尽（be exhaustive），宁可多列不要遗漏
- 隐含待办要说明推理依据，不要凭空编造
- 如果妙记里已经有 AI 生成的待办，也拉出来和自己提取的做交叉对比，确保覆盖全
- 优先执行能力范围内的事，不能执行的标记清楚
- 输出语言跟随用户跟 Agent 对话时使用的语言，保持一致即可
- 不要用破折号（——），用逗号或句号代替
- 不要用破折号（——），用逗号或句号代替
